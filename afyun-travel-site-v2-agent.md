# אפיון מוצר מלא (PRD) — MANTUR · מנוע נסיעות ישראלי

מסמך אפיון מקיף לאתר נסיעות בעברית לקהל ישראלי, שמטרתו לעזור למצוא חופשה משתלמת — בחו"ל או בארץ — ולהרוויח עמלות אפיליאט דרך Travelpayouts. המסמך כתוב כ‑PRD לבנייה ב‑Claude Code.

---

## 0. תקציר מנהלים

מוצר: אתר וצ'אט נסיעות שמתחיל מ**השראה וגמישות** ("יש לי תקציב/תאריך — לאן הכי שווה?") ולא מחיפוש מסלול קבוע. קהל: מטיילים ישראלים. כיסוי כפול: חו"ל (טיסה + מלון ביעד) ותיירות פנים (מלון/צימר בארץ). מודל: אפיליאט (לא OTA) — ההזמנה והתשלום מתבצעים אצל השותף, האתר מקבל עמלה על הזמנה מאושרת.

בידול: לא הגיאוגרפיה אלא ה*מצב* — קונסיירז' גמיש בעברית לישראלי שרוצה "לברוח". כך נמנעים מתחרות חזיתית מול Google Flights על "המחיר הזול למסלול קבוע".

## 1. חזון ומיצוב

- **חזון:** הכלי הראשון שישראלי פותח כשבא לו חופש ואין לו יעד מדויק.
- **הצעת ערך:** "תגידו לי תקציב ותאריך — אני אמצא לכם לאן הכי שווה, בארץ או בעולם."
- **טון:** ישראלי, חם, ענייני, חוסך זמן.
- **כנות:** "הזול ביותר מבין השותפים שלנו", לא הבטחת "הזול בעולם"; מחירי השראה אינדיקטיביים ("החל מ‑~₪X").

### 2.1 HolidayFinder (ישראלי) — מה לאמץ ומה לא

HolidayFinder הוא **OTA** ישראלי (קבוצת Move) שמוכר חבילות טיסה+מלון וגובה תשלום ישירות. לכן חלק מהפיצ'רים שלו אינם ישימים לנו.

| פיצ'ר | לאמץ? | הערה |
|---|---|---|
| חיפוש עם תאריכים גמישים ולוח מחירים | **כן** | מודל מלא: טווח תאריכים + טווח לילות, חודש, "רק סופ"שים", אופי חופשה (6.2) |
| פאנל מסננים עשיר | **כן** | 6.3 |
| קרבה לבית חב"ד / צרכים יהודיים | **כן — ולעשות טוב יותר** | 6.11: דירוג לפי דקות הליכה, לא רק תג |
| עמודי יעד ("חופשה בבודפשט") | **כן** | מנוע ה‑SEO שלנו (6.9) |
| מועדפים / Wishlist | **כן** | 6.7 |
| "החופשה הסודית" — דיל יומי במייל | **כן** | ניוזלטר דילים; מנוע שימור |
| קהילת טלגרם / רשתות | **כן** | ערוץ תנועה |
| מגזין חופשות | **כן** | 6.9 |
| צ'אט קונסיירז' AI | **כן — יתרון שלנו** | להם אין (6.6) |
| Price Lock (נעילת מחיר) | לא | דורש מעמד OTA |
| HolidaySaver (החזר הפרש) / Price Match | לא | דורש מעמד OTA |
| פריסה לתשלומים | לא | דורש סליקה משלנו |
| ניהול הזמנה / ביטול באתר | לא | ההזמנה אצל השותף |

**תובנה:** כמעט כל מה שמושך בחוויית המשתמש נמצא בצד שלנו של הקו. מה שחסום הוא רק שכבת הכסף (תשלום, נעילה, החזר, ביטול).

**מסקנה:** אנחנו בונים את שכבת ה*גילוי וההשראה* (כמו Skyscanner), ומעבירים את ה*הזמנה* לשותף. כל מה שדורש להיות סוחר/OTA (תשלום, נעילת מחיר, נאמנות, מעקב אחרי הזמנה קיימת) — מחוץ ל‑scope.

## 4. עקרונות מוצר

1. **השראה תחילה** — הדף הראשי מזמין לגלות, לא רק לחפש.
2. **גמישות כברירת מחדל** — תאריך/יעד גמישים נתמכים מההתחלה.
3. **Mobile‑first ו‑RTL מלא**.
4. **Handoff מהיר ושקוף** — מינימום חיכוך עד למעבר לשותף, עם תיוג מרקר.
5. **כנות מחיר** — אינדיקטיבי בשלב הגילוי, אמיתי בשלב ההזמנה אצל השותף.
6. **מצבים אופציונליים, ללא עומס** — תכונות לקהלים ספציפיים (כמו מצב דתי/כשר) הן אופט‑אין; מי שלא בחר בהן לא רואה אותן.

## 6. אפיון מסכים

### 6.1 דף הבית
- **מטרה:** להמיר "בא לי חופש" לחיפוש/השראה תוך שניות.
- **רכיבים:** Hero עם בר חיפוש מרובה‑לשוניות; **טוגל חו"ל / בארץ** (בארץ מסתיר את לשונית "טיסות"); בלוק "לאן בא לכם?" (יעדים פופולריים/טרנדיים); בלוק "דילים חמים" (פיד); כניסה בולטת לצ'אט; אמון (מספרי משתמשים/לוגואים של שותפים, גילוי נאות).
- **מצבים:** ריק/טעינה/שגיאה לכל בלוק דינמי.

### 6.2 בר חיפוש ומנוע גילוי
- **לשוניות:** מלונות / טיסות / רכב / דילים.
- **טיסות:** מאיפה, **לאן (כולל "לכל מקום")**, תאריכים **כולל "חודש שלם / החודש הזול"**, נוסעים/מחלקה.
- **מלונות:** יעד, צ'ק‑אין/אאוט, אורחים.
- **רכב:** איסוף, תאריכי איסוף/החזרה.
- **דילים:** מאיפה + "מתי (גמיש)" + תקציב → לא טופס רגיל אלא גילוי.
- **טוגל חו"ל/בארץ:** משנה ברירות מחדל (מטבע, יעדים מוצעים) ומסתיר טיסות במצב "בארץ".

#### מודל התאריכים הגמישים ("מתי?") — רכיב ליבה
שדה התאריכים אינו בורר תאריכים רגיל אלא **אשף גמישות** עם ארבעה מצבים:
1. **תאריכים מדויקים** — הלוך/חזור קבועים (למי שכבר יודע).
2. **טווח תאריכים + מספר לילות** — "יציאה בכל יום בין 18–28 באוגוסט, 5–8 לילות". זהו המצב המרכזי.
3. **חודש שלם** — "אוגוסט", והמערכת מוצאת את השילוב הזול בחודש.
4. **גמיש לגמרי** — "3 החודשים הקרובים / הכי זול".

**בוררי עזר (chips) מעל הבורר:**
- **רק סופ"שים** — מסנן ליציאות חמישי/שישי, 3–4 לילות.
- **רק חופשות/חגים** — מסונכרן ללוח העברי (רלוונטי במיוחד למצב הדתי).
- **משך מהיר** — סופ"ש קצר / שבוע / שבועיים.

**בורר "לאן?" מקביל:** יעד מדויק / קבוצת יעדים (אירופה, יוון והאיים, קפריסין) / **לכל מקום** / **לפי אופי החופשה** (חוף, עיר, טבע, שופינג, רומנטי, משפחתי).

**תוצאה:** מטריצה/פיד של שילובי יעד×תאריך ממוינים לפי מחיר, ולא רשימה אחת — כך המשתמש רואה מיד "מתי הכי זול" ו"לאן הכי זול" יחד.

**ישימות:** מודל "טווח יציאה + טווח לילות" נתמך ב‑Kiwi (בתוך Travelpayouts); "חודש/יום הזול" נתמך בלוח המחירים של Aviasales. אין צורך במעמד OTA.

**אזהרת עלות (קריטי לפני מימוש):** מטריצת יעד×תאריך = ריבוי קריאות מחיר (הרבה צירופי מוצא‑יעד‑תאריך). לפני שמתחייבים למטריצה כחוויית הליבה — לאמת מול מגבלות ה‑rate‑limit, המכסה והעלות של Aviasales/Kiwi ברמת אפיליאט. מיטיגציה: caching אגרסיבי, הגבלת רוחב המטריצה (מספר יעדים/תאריכים בכל שליפה), וטעינה מדורגת.

**התנהגות UX:** ברירת מחדל = מצב 2 (טווח + לילות), כי הוא מייצר את חוויית ה"מצאתי מחיר טוב" החזקה ביותר. מעבר בין מצבים בלי לאבד קלט. הצגת "החל מ‑~₪X" אינדיקטיבי.
- **מצב דתי/כשר (אופציונלי, כבוי כברירת מחדל):** טוגל שמפעיל שכבת חיפוש לשומרי מצוות (ראו 6.11). כשהוא כבוי — החוויה רגילה לחלוטין, בלי שום תוספת.

### 6.3 תוצאות חיפוש
- **פריסה:** רשימת תוצאות + פאנל פילטרים (בצד) + מיון (למעלה).
- **טיסות — פילטרים:** עצירות, חברות, משך, שעות יציאה/חזרה, מחלקה, נמלים סמוכים. מיון: מחיר/משך/המראה.
- **מלונות — פילטרים:** מחיר, דירוג כוכבים, ציון אורחים, מתקנים, מרחק/מיקום, סוג לינה. מיון: מחיר/דירוג/פופולריות.
- **כרטיס תוצאה:** מחיר בולט, פרטי מפתח, תג "דיל" כשמתאים, CTA → handoff מתויג.
- **תוצאות "לכל מקום":** רשימת יעדים מדורגת מהזול ליקר, עם סינון לפי נושא (חוף/עיר/טבע) ולפי תקציב.
- **מצבים:** טעינה (skeleton), אין תוצאות, שגיאת ספק.

### 6.4 עמוד פרט (מלון/טיסה)
- תמונות, מפה, ביקורות (מהשותף), פירוט מחיר ותנאים, מדיניות ביטול (כפי שמסופקת מהשותף), כפתור הזמנה → **handoff מתויג** לאתר השותף. הבהרה: ההזמנה והתשלום מתבצעים אצל השותף.

### 6.5 דילים / השראה
- **Explore לפי תקציב** ("עד ₪X") — יעדים מדורגים.
- **לוח מחירים** (חודש/"התאריך הזול") + תצוגה צבעונית.
- **מפת יעדים זולים** מנקודת מוצא.
- **פיד דילים** (חו"ל ובארץ זה לצד זה).
- **"ירידות מחיר"** — בלוק בהשראת DROPS (מבוסס דאטה/התראות).

### 6.6 צ'אט קונסיירז' (AI)
- קלט חופשי → פענוח מוצא/יעד/תאריכים/תקציב → תשובה קצרה + כפתורי חיפוש מתויגים (חו"ל ובארץ).
- לא ממציא מחירים; מעביר לחיפוש אמיתי.
- מצבי "תפתיעו אותי" ו"יש לי ₪X, לאן?".
- backend מאובטח (סעיף 10).

### 6.7 המסעות שלי — Wishlist + התראות מחיר
- שמירת חיפושים/יעדים (כמו Trips).
- **התראת מחיר**: המשתמש "עוקב" אחרי מסלול/יעד ומקבל מייל/פוש כשהמחיר יורד. בגרסה האפיליאטית — מבוסס נתוני חיפוש שלנו, וההזמנה בלינק מתויג. (ללא נעילת מחיר — זה OTA בלבד.)

### 6.8 חשבון משתמש
- התחברות (אימייל/חברתי), העדפות (מוצא ברירת מחדל, מטבע, נושאים), ניהול התראות, היסטוריה.
- אופציונלי ל‑MVP; אפשר להתחיל ללא חשבון ולהוסיף בהמשך.

### 6.9 עמודי יעד (SEO) ומגזין
- מדריך יעד: מתי לטוס, כמה עולה, מה לעשות, + טופס חיפוש מוטמע + טבלת "הטיסות/מלונות הזולים ל…".
- מגזין: תוכן השראה ("5 בריחות סופ"ש מתל אביב עד ₪1500") עם קישורים מתויגים.
- חשוב: White Label אינו אינדקסבילי → התנועה האורגנית מגיעה מעמודי התוכן האלה, לא מ‑WL.

### 6.10 לוח בקרה לסוכן (Agent Dashboard)
- **מטרה:** לתת לסוכן תמונת ביצועים מלאה — כמה נכנסו, כמה חיפשו, כמה לחצו וכמה סגרו — במקום אחד.
- **גישה:** אזור מוגן (`/admin`) מאחורי התחברות; נגיש רק לסוכן. (אופציה עתידית: כמה סוכנים, כל אחד רואה רק את שלו.)
- **רכיבים:**
  - כרטיסי KPI: מבקרים, חיפושים, קליקים לשותף, הזמנות, עמלה (מאושר + ממתין), עם שינוי מול תקופה קודמת.
  - משפך המרה: מבקרים → חיפושים → קליקים → הזמנות, עם אחוז המרה בין שלבים.
  - מגמה לאורך זמן עם בורר טווח (7 / 30 / 365 יום).
  - יעדים מובילים ומקורות תנועה (איזה ערוץ מביא הזמנות: SEO / צ'אט / דילים / חברתי).
  - פירוק לפי קטגוריה (טיסות מול מלונות מול רכב — מבליט שהמלונות מרוויחים יותר).
  - טבלת הזמנות אחרונות עם סטטוס (ממתין/מאושר/בוטל).
- **מקורות נתונים:** מדדי מבקרים/חיפושים/קליקים/שיחות צ'אט הם של האתר ובזמן אמת; נתוני ההזמנות והעמלה מגיעים מ‑Travelpayouts לפי SubID ומתעדכנים עם עיכוב (אישור רק אחרי מימוש ההזמנה) — לכן העמלה מוצגת מפוצלת ל"מאושר" ו"ממתין".
- **מצבים:** טעינה, אין נתונים בטווח, שגיאת סנכרון.

### 6.11 מצב דתי/כשר (אופציונלי)
- **עיקרון:** כבוי כברירת מחדל. המשתמש מפעיל "מצב כשר/שומר‑שבת" (טוגל בבר החיפוש או בהעדפות), ורק אז נוספות היכולות הבאות. ללא הפעלה — האתר רגיל לגמרי, בלי עומס ויזואלי.
- **עוגן חיפוש:** בחירת עוגן יהודי — בית חב"ד / בית כנסת מסוים / שכונה יהודית / אשכול מסעדות כשרות — והמלונות נשלפים ברדיוס סביבו.
- **דירוג לפי הליכה:** התוצאות מדורגות לפי מרחק הליכה לחב"ד/שול הקרוב ("X דק' הליכה"), כי בשבת לא נוסעים.
- **פילטרים ייעודיים:** מרחק הליכה מקסימלי לשול/חב"ד, מעלית שבת, מפתח מכני, ארוחת בוקר/מטבח כשר + רמת השגחה, צ'ק‑אין לפני שבת / צ'ק‑אאוט במוצ"ש, עירוב, מקווה בקרבת מקום, תוכניות פסח/סוכות.
- **טיסות:** בקשת ארוחה כשרה (KSML), והתראה/סינון על טיסות שממריאות/נוחתות בשבת או סמוך לכניסתה.
- **לוח עברי:** זמני הדלקת נרות, כניסת/יציאת שבת וחגים ביעד.
- **שכבות נתונים (מופעלות רק במצב זה):** Google Places לבתי כנסת/מסעדות כשרות/חב"ד; שכבת אוצרות ידנית למלונות כשרים/שומרי‑שבת (התכונות אינן בנתוני ה‑OTA); HebCal ללוח השבת. (Chabad Locator — בכפוף להרשאה; לא לגרד.)
- **שקיפות:** הבהרה קבועה "אמתו כשרות וסידורי שבת מול המלון", כי הסטטוס משתנה.
- **התמדה:** בחירת המשתמש נשמרת (העדפות/חשבון), כך שלא צריך להדליק בכל ביקור.
- **במה נהיה חזקים יותר מהמתחרים** (שמסתפקים בתג "קרוב לחב"ד"):
  - **דירוג מספרי בדקות הליכה** לכל תוצאה, לא תג בינארי — "7 דק' הליכה לחב"ד".
  - **מפת שבת** ליעד: המלון, החב"ד/שול, מסעדות כשרות, מקווה ותחום העירוב במבט אחד.
  - **בדיקת התאמה לשבת בטיסה:** אזהרה אוטומטית אם ההמראה/הנחיתה קרובה מדי לכניסת שבת ביעד.
  - **חבילת "שבת ביעד":** לכל יעד — זמני כניסה/יציאה, מניינים בקרבת מקום, ומלונות מדורגים לפי הליכה.
  - **צ'אט מודע הלכתית:** "צריך מלון בפראג במרחק הליכה מחב"ד לשבת" → תשובה עם מרחקים וזמני שבת.

## 7. זרימות משתמש

1. **השראה→הזמנה:** בית → "לכל מקום"/דילים → בחירת יעד → תוצאות → handoff.
2. **חיפוש ישיר:** בר → תוצאות → פילטרים → handoff.
3. **צ'אט:** בקשה חופשית → הצעות מתויגות → handoff.
4. **בארץ:** טוגל בארץ → מלון/צימר → handoff (ללא טיסה).
5. **מעקב:** עוקב אחרי יעד → התראת ירידת מחיר → חוזר → handoff.
6. **תוכן→חיפוש (SEO):** מדריך יעד אורגני → טופס מוטמע → handoff.

## 10. ארכיטקטורה טכנית

- **Frontend:** Next.js/React (SSR לעמודי תוכן אינדקסביליים) + קומפוננטות אינטראקטיביות; RTL; אירוח (Vercel/דומה).
- **Travelpayouts:** חשבון → Project → חיבור פר‑תוכנית → כלים (ווידג'טים/טפסים/לינקים). בניית deep‑link עם `marker` + `locale=he` + `currency=ils`. WL (אם בכלל) על **תת‑דומיין**. SubID פר‑מיקום למדידה.
- **Backend לצ'אט:** endpoint `/api/chat`; **מפתח Anthropic בצד שרת בלבד**; מודל `claude-sonnet-5` (הדור הנוכחי; $3/$15 למיליון טוקנים, מחיר היכרות $2/$10 עד 31.08.2026). אופציית חיסכון: `claude-haiku-4-5` ($1/$5) לפענוח בקשות פשוטות — לבחור לפי בדיקת איכות על שאילתות אמיתיות.
  - **פענוח מובנה:** Structured Outputs (`output_config.format` עם json_schema, או `messages.parse`) — פלט JSON מובטח‑סכמה `{origin, destination, dates, budget, scope}` במקום פרסינג טקסט שביר.
  - **חשיבה (thinking):** ב‑Sonnet 5 חשיבה אדפטיבית דולקת כברירת מחדל — לצ'אט מהיר וזול להגדיר `thinking: {type: "disabled"}` או `effort: low`.
  - **Prompt caching:** system prompt קבוע לחלוטין (בלי תאריך/מזהה משתנה בתחילתו — כל שינוי בייט שובר את המטמון) + `cache_control`; קריאות מהמטמון ≈10% מהמחיר. זהו מנגנון בקרת העלות המרכזי.
  - **Streaming:** תגובות הצ'אט נשלחות בזרימה (טוקן‑טוקן) לחוויית צ'אט תקינה.
  - **אבטחת הצ'אט:** rate‑limit + מגבלת אורך קלט + תקרת הוצאה חודשית קשיחה + הגנה מ‑prompt injection (הבוט מוגבל לתחום נסיעות בלבד ב‑system prompt, ונבדק על כך).
- **שכבת דאטה (שלב מתקדם):** API מחירים (למשל Aviasales Data) ללוח/דילים; caching אגרסיבי לשמירה על יחס חיפוש‑להזמנה ועלויות.
- **תשתית התראות:** איחסון מסלולים במעקב + ג'וב תקופתי שבודק מחיר ושולח מייל/פוש.
- **אנליטיקס:** GA4/Plausible + סטטיסטיקת Travelpayouts; מיפוי SubID למקור תנועה.
- **לוח בקרה לסוכן:** אזור מוגן באימות; קליטת אירועי first‑party (מבקר/חיפוש/קליק/צ'אט) + ג'וב שמסנכרן הזמנות ועמלות מ‑Travelpayouts לפי SubID; הצגה מצרפית בלוח.
- **מצב דתי/כשר (אופציונלי):** שכבת POI (Google Places) + שכבת אוצרות מלונות כשרים + HebCal ללוח השבת; חישוב מרחק הליכה בין מלון לשול/חב"ד. מופעל רק כשהמשתמש מדליק את המצב.

## 17. סיכונים ומיטיגציות

- **עמלת טיסה דקה** → דגש על מלונות/השלמות; מדידת EPC.
- **מיצוב רחב** → לחדד טון/נישה ("בריחות מ‑TLV בתקציב").
- **לא תמיד הזול ביותר** → ניסוח כן, איחוד מקורות שותפים, מחירי גילוי אינדיקטיביים.
- **עלות API לצ'אט/דאטה** → caching, בוט רזה, תקרת הוצאה.
- **תלות בתנאי Travelpayouts** → תאימות תנועה, גיוון מקורות.
- **SEO איטי להבשלה** → לתכנן ערוץ תנועה נוסף לטווח קצר.
- **אוצרות מצב הכשר — סיכון מספר 1 של הפיצ'ר** → מלונות כשרים/שומרי‑שבת ונתוני POI (בתי כנסת/חב"ד/מקווה/עירוב) אינם קיימים בפיד ה‑OTA ודורשים אוצרות ידני מתמשך (עלות תפעולית, לא רק פיתוח). גבולות עירוב במיוחד קשים כדאטה. מיטיגציה: **לפַיילט על 2–3 יעדים** ולוודא איכות אוצרות לפני הבטחת כיסוי רחב; שקיפות "אמתו מול המלול".
- **עלות/rate‑limit של מטריצת התאריכים** → הפיצ'ר הדגל עלול לצאת איטי/יקר; לאמת מול ה‑API האמיתי לפני התחייבות (ראו 6.2).
- **כיסוי צימרים ישראלים דל אצל השותפים** → לבדוק מלאי בפועל (Booking/Hotellook) לפני בניית הטוגל "בארץ"; לשקול שותף מקומי משלים.
- **תמיכת `locale=he` לא מובטחת בכל תוכניות Travelpayouts** → לוודא פר‑תוכנית שעמוד הנחיתה בעברית; handoff שנוחת באנגלית שובר את חוויית הקונסיירז' ברגע ההמרה.
